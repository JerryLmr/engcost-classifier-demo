## Decisions
- 当前阶段优先保证演示稳定性和可解释性，不引入训练集、数据库或复杂检索。
- 保留“规则优先 + LLM 兜底”的策略，用于平衡稳定性与覆盖率。
- 降级默认分类暂定为“公共设施 / 公共区域维修”，后续可按业务反馈调整。
- 针对 `单元门` 触发过强的问题，当前决定先归档业务口径，不立即修改规则，避免在全量样本稳定后继续过拟合。
- `单元门 + 粉刷/翻新/涂装/饰面` 默认不归 `门禁设施`，优先按 `楼道装修` 处理；若现有体系不便承接，可退到 `公共设施 / 公共区域翻新`。
- `单元门 + 门禁/门控/刷卡/人脸/自动门/防盗门 + 更换/维修/升级` 归 `门禁设施`；`单元门` 单独出现不应视为强门禁触发词。

## Known Limitations
- 分类体系和规则仍以 Python 常量形式维护，尚未迁移到 JSON 或数据库。
- 当前规则未经过系统样本回归，准确率仍需真实工程名称样本验证。
- 前端 `API_BASE` 仍默认指向 `http://127.0.0.1:8000`，跨环境部署时需要显式配置。
- 当前“待人工确认/低置信度”仍未对外暴露，规则不确定时仍只能走 LLM 或默认降级。
- API 自动化测试在缺少 `httpx` 时会被跳过，测试依赖仍需补齐。
- 当前复合工程检测仍是启发式规则，适合先看效果，不代表已完成最终业务口径。
- `structure_type` 已对外暴露，但中文展示映射尚未在前端结果区实现。
- `单元门` 相关边界当前仅完成业务归档，尚未做规则层最小收敛；若后续出现重复、成批误判，再按最小改动实施。
