# PROJECT STATE

## Current Milestone
- Demo 已完成从“物业工程名称输入”到“一级/二级分类输出”的稳定闭环，支持单条分类、Excel 批量分类、结果分析与前端展示。

## System Capabilities
- 支持单条工程名称分类，返回一级分类、二级分类、分类方式、分类依据、结构类型、复合工程相关字段。
- 支持 Excel 批量分类，自动追加分类结果、复合工程标记和复核相关列并下载结果文件。
- 支持“规则优先 + LLM 兜底”的分类策略，并在体系外场景下提供默认分类回退。
- 支持复合结构识别，区分 `single_project`、`multi_system_same_domain`、`composite_project`。
- 支持基于目录的批量回归处理、结果对比和结果分析，并可导出 Excel 汇总报表。
- 支持规则双轨来源：默认使用 JSON 配置规则，Python 基线规则保留为对照与回退路径。
- 前端支持上传已分类结果文件并展示摘要统计、分类分布、异常/复杂样本。

## Recent Changes
- 已完成规则从 Python 常量到 JSON 配置的迁移，并验证 JSON 与 Python 基线输出字段级 0 差异。
- 已将默认规则来源切换为 JSON，同时保留 Python 规则用于回归对比。
- 已新增结果分析服务与 `/api/analyze-excel` 接口，前端可直接分析已分类结果 Excel。
- 已优化前端结果说明，统一展示文案，并补充结构类型、复合原因、候选分类、异常样本筛选和排序能力。
- 已对 `单元门`、`公共区域粉刷/翻新` 等高频误判边界做最小收敛，当前全量样本结果保持稳定。
- 已统一“建议复核”口径：体系外默认分类、同域多系统、复合工程样本默认建议人工复核，分析接口对历史结果文件也按新口径重算。
- 已新增本地批量分类、结果对比、目录级分析脚本，支持回归和结果验收。

## Decisions
- 当前阶段以“演示稳定、结果可解释、规则可维护”为优先目标，不引入训练集、数据库或复杂检索。
- 分类策略继续保持“规则优先 + LLM 辅助分类 + 体系外默认分类”。
- 规则配置默认以 JSON 为准；Python 规则暂保留，作为基线对照与排障回退路径。
- 复合工程、建议复核、结构类型继续保留为解释字段；其中“建议复核”统一按 `体系外默认分类 / 同域多系统 / 复合工程` 三类触发，不再区分复合工程主次是否明显。
- 剩余业务边界问题暂不继续细调，待后续与领导沟通后再决定是否调整分类体系。

## Known Limitations
- 当前分类体系仍是项目内自行定义，个别样本是否属于“体系外”依赖业务口径，不代表分类体系最终定稿。
- 规则准确率已通过当前样本集验证，但仍需靠后续新样本持续回归，而非一次性证明长期完备。
- 仍存在少量需要业务确认的边界，例如 `楼宇对讲 / 门禁 / 弱电系统`、`道路工程 / 停车交通`。
- 当前全量样本中仍有少量 `体系外默认分类`，例如幕墙玻璃补漏打胶类样本，暂未单独扩类。
- 前端 `API_BASE` 仍默认指向本地 `http://127.0.0.1:8000`，跨环境部署时需要显式配置。
- API 自动化测试在缺少 `httpx` 时会跳过，测试环境依赖仍需补齐。

## Next Steps
- 用后续新增真实样本继续验证 JSON 规则版本的稳定性。
- 与领导确认剩余业务边界后，再决定是否调整分类体系或继续细化规则。
- 若分类体系稳定，下一阶段可考虑提升配置维护体验，例如配置校验、可编辑流程或简单管理界面。
- 若前端继续增强，可在分析区增加排序、筛选或详情展开，而不是继续堆叠更多规则说明。
