# PROJECT STATE

## Current Milestone
- 项目已形成“工程分类 + 维修资金审计”双链路基础闭环：分类能力稳定可用，`/api/audit` 已能输出总审计结果，当前阶段重点转向补全完整审计流程所需的结构化输入与分项审计输出。

## System Capabilities
- 支持单条工程名称分类，返回一级分类、二级分类、分类方式、分类依据、结构类型与复核相关字段。
- 支持 Excel 批量分类与结果分析，前端可展示摘要统计、分类分布、异常/复杂样本。
- 支持基于 `object_catalog.json + rule_mapping.json + rule_engine.json` 的维修资金审计基础闭环。
- 支持 `/api/audit` 输出总审计结果，并保留对象映射、标签、原因码、依据、审计路径等可解释字段。
- 支持结构化审计输入分组与分项审计输出骨架，为后续 OCR / 文档解析和完整流程审计预留接口。

## Recent Changes
- 已新增 `/api/audit` 基础链路，打通对象映射、标签映射、规则裁决和统一输出。
- 已将审计主规则迁移到 JSON 规则引擎，移除关键结论对代码硬编码的依赖。
- 已补充正常维修默认补件规则与弱灰区转正常流程规则，审计结果解释更具体。
- 已扩展审计输入 schema，支持使用范围、流程、资料、时序、金额、应急等结构化事实分组。
- 已新增分项审计输出骨架和 OCR 目标字段容器，为后续文档解析接入做准备。

## Decisions
- 当前审计链继续以“规则引擎主导”为原则，LLM 与 OCR 仅承担抽取和解释，不承担最终裁决。
- `/api/classify` 与 `/api/audit` 并行保留；分类接口不作为最终审计依据。
- `/api/audit` 继续沿用单一路径做兼容扩展，不新增 `v2` 接口。
- 分项审计结果与总结果统一使用 `compliant / non_compliant / need_supplement / manual_review` 四类结果；不适用通过 `applicable=false` 表达。
- `PROJECT_STATE.md` 继续保持快照性质，不作为历史日志使用。

## Known Limitations
- 当前审计规则仍以一期最小闭环为主，分项审计规则骨架已建立，但细粒度规则尚未全部补齐。
- OCR / 文档解析仅完成目标字段与数据结构预留，尚未接入真实文件上传和抽取流程。
- 审计链前端展示仍保持最小可用状态，完整补件表单与分项结果展示尚未展开。
- 目录映射与标签推断仍存在部分代码兜底逻辑，后续仍需逐步下沉到配置层。
- API 自动化测试在缺少 `httpx` 时会跳过，测试环境依赖仍需补齐。

## Next Steps
- 继续补全分项审计规则，使使用范围、流程、资料、时序、金额、应急六类结果具备更强判定能力。
- 将 OCR / 文档解析的抽取结果接入审计上下文，减少人工补录字段。
- 按业务口径补充更多结构化输入字段与规则资产，并持续做回归验证。
- 在后端结构稳定后，再决定是否扩展前端分项审计展示与补件交互。
